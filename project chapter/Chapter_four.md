## 4.1 Programming Language of Implementation
The implementation of MoveTogether involved a single primary programming language, JavaScript, deployed across all layers of the system architecture. JavaScript was used on the backend through the Node.js runtime environment to power the Express API server, handle authentication logic, manage trip operations, and coordinate real-time communication through Socket.io. The same language was used on the frontend through the React library to build the entire user interface, including the dashboard, trip creation forms, real-time chat, and profile management pages. This full-stack JavaScript approach is commonly referred to as the MERN stack, which stands for MongoDB, Express, React, and Node.js. The database layer relied on MongoDB, a NoSQL document database, accessed through the Mongoose Object Data Modelling library, which provided schema definitions and query abstractions for all database operations.

## 4.1.1 Justification of Programming Languages Used
JavaScript was selected as the sole programming language for this project for three reasons that are directly tied to the requirements of the MoveTogether platform.

First, using JavaScript on both the client and server eliminates the impedance mismatch that occurs when different languages are used at different layers of the stack. The Trip object schema defined in the Mongoose model on the backend uses the same field names and data structures that the React frontend expects when rendering trip cards on the dashboard. This consistency reduced the time spent translating data between layers and prevented the serialisation bugs that commonly occur in polyglot architectures (Tilkov and Vinoski, 2016).

Second, the real-time communication requirements of the trip chat feature demanded a language with strong event-driven capabilities. Node.js is built on Google's V8 engine and operates on a single-threaded, non-blocking event loop architecture that handles thousands of concurrent WebSocket connections without spawning new threads for each connection (Tilkov and Vinoski, 2016). This made it the natural choice for the Socket.io-based trip chat system, where multiple users in the same trip room must receive messages instantly without polling the server.

Third, the React library was selected for the frontend because its component-based architecture allowed the eleven main pages of the application to be developed as independent, reusable units that share a single authentication state. These pages include the Authentication Page, Dashboard, Explore, Create Trip, Edit Trip, Trip Detail, Messages, Profile, Public Profile, Admin Dashboard, and the Password Reset flow. React's virtual Document Object Model rendering model ensures that only the components affected by a state change are re-rendered (Osmani, 2023), which is important for the Explore page where trip cards, filter states, and cost calculations must update dynamically as users interact with the interface.

MongoDB was selected as the database because its document-based storage model maps directly to the JSON objects used throughout the JavaScript stack. Trip documents containing nested arrays of traveler references and coordinate sub-documents can be stored and retrieved without the object-relational mapping overhead that a relational database would require. Mongoose provides schema validation at the application layer, which gives the flexibility of a document database while maintaining the data integrity constraints needed for a multi-user coordination platform (Banker *et al.*, 2016).

## 4.3 Implementation Guidelines
The implementation of MoveTogether followed the five-sprint Agile development plan outlined in Chapter Three. This section documents the key implementation steps taken during each sprint, providing a traceable account of how the system was built from an empty project directory to a fully functional group coordination platform.

Sprint One covered the project foundation. The project was initialised by creating a root directory with separate client and server folders. The Express server was configured in server.js to listen on port 5000 and connect to a MongoDB Atlas instance using the connection string stored in an environment variable. The User model was built using Mongoose with fields for name, email, hashed password, bio, phone, role, email verification status, average rating, and emergency contact information. The authentication system was implemented with five API endpoints covering registration, login, email verification, code resend, and the authenticated user retrieval route. Password hashing was implemented using bcryptjs with a salt factor of 10, and JWT tokens were configured to expire after seven days. A six-digit email verification code system was built to confirm user identity upon registration. The React frontend was initialised using Vite, and the Authentication Page was built with separate Login and Register forms that communicate with the backend through Axios.

Sprint Two covered the trip creation and discovery modules. The Trip model was built with fields for creator, title, origin, destination, meeting point, departure date, transport type (Bus, Car, Train, Flight, Walk, Bike, or Other), category (Transport, Fitness, Adventure, Events, Social, or Other), maximum travelers, total cost, status, and coordinate sub-documents for the origin, destination, and meeting point locations. The trip creation controller was implemented with comprehensive input validation that checks field lengths, date validity, transport type enumeration, and cost constraints. The Multer middleware was integrated to handle trip image uploads, storing files in the server/uploads directory. The Explore page was built on the frontend with a filtering system that supports keyword search, category filtering, transport type filtering, date filtering, status filtering, and sorting by newest, soonest departure, or almost-full trips. The Create Trip page was built with form fields for all trip attributes and an integrated Location Picker component built on top of Leaflet and React-Leaflet that allows users to select origin, destination, and meeting point coordinates on an interactive map.

Sprint Three covered the trip joining flow and the cost-splitting mechanism. The join request system was implemented as a two-step process where a user first sends a join request, and the trip creator then accepts or rejects the request. When a request is accepted, the backend moves the user from the pendingRequests array to the travelers array and recalculates the per-person cost by dividing the total cost by the current number of travelers. If the travelers array reaches the maxTravelers threshold, the system automatically updates the trip status from "active" to "full". The trip creator can also remove travelers from the trip, which reverts the status back to "active" if it was previously "full". Email notifications were integrated using Nodemailer to alert the trip creator when a new join request is received, and to alert the requesting user when their request is accepted. An in-app notification system was also built using a dedicated Notification model that stores notification type, message, recipient reference, and read status. Real-time notification delivery was implemented through Socket.io, where each user joins a personal room identified by their user ID, and the server emits a "new-notification" event whenever a notification is created.

Sprint Four covered the real-time chat system and the user interaction features. The TripMessage model was built to support text messages and location sharing, with fields for trip ID, sender reference, text content, latitude, and longitude. The trip chat controller restricts message sending to trip participants only and blocks messages in completed or cancelled trips. The TripChat component on the frontend establishes a Socket.io connection to a room identified by the trip ID, and incoming messages are rendered instantly without requiring a page refresh. The direct messaging system was built separately using a Message model that stores sender, receiver, conversation ID (generated by sorting and concatenating the two user IDs), text content, and read status. The Messages page on the frontend displays a conversation list and a message thread side by side. The Rating system was implemented to allow users to rate other participants after a trip is completed, with a five-point scale and an optional comment field. A unique compound index on tripId, reviewerId, and reviewedUserId prevents duplicate ratings for the same user on the same trip.

Sprint Five covered the admin panel, profile management, and system-wide integration. The Admin Dashboard was built with endpoints for viewing all users, suspending and unsuspending user accounts, viewing user reports, and managing trip statuses. The Report model was implemented to allow users to report other users for reasons including Spam, Harassment, Inappropriate Content, Safety Concern, or Other, with a description field and a status that tracks whether the report is pending, resolved, or dismissed. The Profile Page was built to display and edit the user's name, bio, phone number, and emergency contact information, with input validation enforced on both the frontend and backend. The Forgot Password and Reset Password flows were implemented using a cryptographically random token stored in the database with a one-hour expiry window. The trip status management system was finalised with a state machine that enforces valid transitions. An active trip can move to full, confirmed, or cancelled. A full trip can move to active, confirmed, or cancelled. A confirmed trip can only move to completed or cancelled. Completed and cancelled trips are terminal states that cannot be changed. Email notifications are sent to all travelers when a trip is confirmed, cancelled, or completed.

## 4.4 Results and Discussion
This section presents the results produced by MoveTogether across its core functional modules and discusses each result in relation to the objectives stated in Chapter One and the findings of related literature reviewed in Chapter Two. The system was tested using sample data created by multiple test user accounts operating within a simulated university campus environment.

[Figure 4.1: Authentication Page showing the Registration form with email and password fields]

Figure 4.1 shows the authentication interface of MoveTogether. The page presents two switchable forms for Login and Registration. The registration form requires a name, email address, and password. The password field enforces a complexity requirement that demands at least eight characters, one uppercase letter, one lowercase letter, one digit, and one special character. Upon successful registration, the system generates a six-digit verification code and redirects the user to the Check Email page for verification. This multi-step authentication process addresses the trust concern identified by Adegoke (2023), who found that 67% of respondents cited trust as a significant barrier to informal ride-sharing. By requiring verified email addresses and enforcing strong passwords, MoveTogether establishes a baseline level of user accountability that informal WhatsApp coordination cannot provide.

[Figure 4.2: Dashboard showing the user's active and upcoming trips with per-person cost calculations]

Figure 4.2 shows the main dashboard of MoveTogether after a user logs in. The dashboard displays all trips that the user has created or joined, sorted by departure date. Each trip card shows the trip title, origin, destination, departure date, number of travelers relative to the maximum capacity, and the dynamically calculated per-person cost. The per-person cost is computed on every request by dividing the total cost by the number of current travelers. This means the cost displayed on the dashboard automatically decreases as more travelers join the trip, which provides a direct visual incentive for users to share trips with others. This dynamic cost calculation directly addresses the first objective stated in Chapter One, which was to design and develop a platform that enables users to create and join group-based trips. The dashboard also confirms the finding of Kim *et al.* (2019), who demonstrated that participants in group ordering conditions spent 35% less on average compared to individual ordering.

[Figure 4.3: Explore page showing filtered trip listings with search, category, and sort options]

Figure 4.3 shows the Explore page, which serves as the trip discovery interface. Users can search for trips by keyword, filter by category (Transport, Fitness, Adventure, Events, Social), filter by transport type (Bus, Car, Train, Flight, Walk, Bike), filter by date, and sort results by newest, soonest departure, or almost-full. Each trip card displays a fill ratio indicator that shows how close the trip is to reaching its maximum capacity. This structured discovery mechanism directly addresses the limitation of WhatsApp identified in Chapter Three, where users had to manually scroll through long chat histories to find relevant coordination opportunities. Hossain and Rahman (2020) found that 79% of WhatsApp group users experienced information overload when trying to coordinate shared services. The filtered, searchable interface of MoveTogether eliminates this problem by presenting only the trips that match the user's specific criteria.

[Figure 4.4: Trip Detail page showing trip information, travelers list, join request button, and cost split]

Figure 4.4 shows the Trip Detail page for an active trip. The page displays the full trip information including the origin, destination, meeting point, departure date, transport type, category, total cost, per-person cost, and a list of current travelers. Users who are not yet part of the trip can click the "Request to Join" button, which sends a notification to the trip creator. The trip creator sees pending requests and can accept or reject each one individually. When a request is accepted, the system adds the user to the travelers array, recalculates the per-person cost, and checks whether the maximum capacity has been reached. If the capacity threshold is met, the trip status automatically changes from "active" to "full". This threshold-based status management directly addresses the second objective stated in Chapter One, which was to implement a group formation mechanism that allows activities to proceed only when a specified number of participants is reached. The two-step join process (request followed by creator approval) also addresses the trust barrier identified by Adegoke (2023) and Eze *et al.* (2021), who both found that unknown participants were a major concern in informal coordination.

[Figure 4.5: Trip Chat interface showing real-time messages and location sharing between trip participants]

Figure 4.5 shows the Trip Chat interface for a confirmed trip. The chat panel is accessible from the Trip Detail page and uses Socket.io for real-time message delivery. Messages appear instantly for all participants without requiring a page refresh. The chat also supports location sharing, where a user can send their current coordinates as a clickable map pin. This allows trip members to share meeting point updates or track each other's proximity before departure. The chat is restricted to trip participants only, and the system blocks message sending in completed or cancelled trips to maintain a clean record. This real-time coordination feature directly addresses the third objective stated in Chapter One, which was to design and implement coordination features that enable group members to agree on meeting points and activity execution plans. The finding of Eze *et al.* (2021) that 73% of students experienced difficulties with informal WhatsApp coordination is directly addressed by this purpose-built chat that is scoped to a specific trip context rather than a noisy general-purpose group chat.

[Figure 4.6: Profile Page showing user information, emergency contact, and average rating]

Figure 4.6 shows the Profile Page where users can view and edit their personal information. The page displays the user's name, bio, phone number, email verification status, emergency contact details, and their average rating from completed trips. The system requires users to complete their profile (including bio, phone, verified email, and emergency contact) before they are allowed to create a trip. This profile completeness requirement ensures that every trip creator has provided sufficient identification information for other users to feel confident joining their trip. The rating system, which allows users to rate their co-travelers on a scale of 1 to 5 after a trip is completed, provides a feedback mechanism that incentivises good behaviour and helps build community trust over time. Tauscher and Laudien (2018) found that cost-splitting applications were used exclusively within pre-existing social groups. MoveTogether extends this by using verified profiles and ratings to build trust between strangers, enabling coordination beyond existing social circles.

[Figure 4.7: Admin Dashboard showing user management, trip oversight, and report handling]

Figure 4.7 shows the Admin Dashboard, which provides platform-level oversight capabilities. The admin can view all registered users, suspend or unsuspend accounts, review user reports submitted by other users, and manage trip statuses. The report system allows users to flag other users for Spam, Harassment, Inappropriate Content, Safety Concern, or Other reasons, with a mandatory description field. Each report has a status that tracks whether it is pending review, resolved, or dismissed. This administrative layer provides the governance structure needed to maintain platform safety and user trust, which is particularly important in a system that facilitates physical meetups between users who may not know each other prior to joining a trip.

Taken together, these results demonstrate that MoveTogether successfully achieves all five objectives stated in Chapter One. The system enables users to create and join group-based trips across multiple service categories, implements threshold-based group formation with automatic status management, provides real-time coordination features through the trip chat with location sharing, offers automated equal-split cost calculation that dynamically adjusts as participants join or leave, and delivers a usable platform that was evaluated through functional testing across all core modules. Compared to the informal WhatsApp coordination method analysed in Chapter Three, MoveTogether eliminates manual cost calculation, provides structured trip discovery, automates threshold management, and maintains clear records of participation and financial commitments. These improvements directly address the coordination fatigue problem documented by Akinwale and Omole (2022), who found that 63% of students were discouraged from regularly organising group activities due to the effort required when using informal messaging tools.
